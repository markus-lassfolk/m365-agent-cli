# Feature Roadmap

Tracking document for the post-2026.7.4 feature push. Generated from a review of the
codebase's agent-usability gaps and coverage holes. Status is updated as each item lands;
each landed item references the commit(s) that implemented it.

Legend: ⬜ Not started · 🟡 In progress · ✅ Done · ⏭️ Deferred (needs live Exchange / external
verification / a product decision)

## Top picks (agent-usability infrastructure)

| # | Feature | Status | Notes |
|---|---------|--------|-------|
| 1 | `describe` — machine-readable command/option manifest | ✅ | `m365 describe` (full JSON manifest), `--list` (fast top-level overview), `--command "rules create"` (scoped lookup). `src/lib/command-manifest.ts` + `src/commands/describe.ts`. |
| 2 | Native MCP server mode | ✅ | `m365 mcp` starts a stdio JSON-RPC MCP server (`src/lib/mcp-server.ts`, `src/commands/mcp.ts`). Reflects #1's `describe` manifest into one MCP tool per leaf command (JSON schema built from that command's own arguments/options — repeatable-option detection covers both Commander's native `<x...>` variadic syntax and this codebase's more common accumulator-function + array-default convention). A `tools/call` executes by spawning `bun src/cli.ts <argv>` — the same entry point a human runs — rather than re-implementing command logic in-process (which would've required fighting Commander's per-command `Option` state being shared across the module-singleton `Command` instances every command file exports). `mcp`/`serve`/`login` are excluded from the tool list (self-referential / long-running / interactive). Covered by unit tests (mocked `runCli`) plus a real-subprocess end-to-end test that speaks actual JSON-RPC over stdio to a spawned `bun src/cli.ts mcp` process. |
| 3 | `--dry-run` for mutations | ✅ | Transport-level: `callGraphAt`/`callEws` halt and print the resolved request instead of sending it, gated by `M365_DRY_RUN` (synced from the root `--dry-run` flag via a Commander `preAction` hook). Works uniformly for all ~60 commands with no per-command wiring. For multi-step flows, only the first mutating request is shown. `src/lib/dry-run.ts`. |
| 4 | Output shaping: `--fields` projection + `--ndjson` streaming | ✅ (scoped rollout) | `src/lib/output-shape.ts`: `--fields <dot-paths>` client-side projects each row (silently omitting missing paths — payloads aren't uniform); `--ndjson` prints one compact JSON object per row instead of one pretty array. Named `--fields` (not `--select`) since `--select` already means server-side OData `$select` passthrough on 15+ commands. Unlike `--dry-run`/`--cache`, there is no single transport-layer choke point every list command funnels through (692 `console.log(JSON.stringify(...))` call sites across 54 command files, non-uniform envelopes) — so this is a per-command opt-in, wired end-to-end (options, both EWS and Graph backends, tests, docs) into `mail`'s list view as the reference implementation. Other high-volume list commands (`outlook-graph list-messages`/`messages-delta`, `todo`, `planner list-tasks`, `find`, `files`) are natural follow-ups using the same helper. |

## Reliability & safety

| # | Feature | Status | Notes |
|---|---------|--------|-------|
| 5 | Structured error envelope in `--json` mode | ✅ | `{ error: { message, code?, status?, retriable?, requestId? } }` (mirrors Graph's own error shape) instead of a bare string. `toJsonError()` in `src/lib/json-error.ts` normalizes any input (GraphError/OwaError object, string, Error, undefined); applied at all ~168 `--json` error print sites across 21 command files. Sites that already had the full error object (not just its pre-extracted `.message`) get full `code`/`status`/`requestId` fidelity; sites that only ever held a message string still get the consistent shape but without those extra fields — a follow-up could thread the full error object through more call sites for richer fidelity everywhere. |
| 6 | Auto-chunking Graph `$batch` helper | ✅ | `graphBatchAll()` in `src/lib/graph-advanced-client.ts`: accepts any number of sub-requests, splits into `≤20`-request chunks (`chunkGraphBatchRequests`), POSTs each chunk to `/$batch` sequentially via the existing `graphPostBatch`, and merges `responses` back into one array in request order. Validates unique `id`s and that any `dependsOn` chain stays within a single chunk before sending anything (Graph requires dependents to share a batch). Wired into the `graph batch` CLI command, replacing its old hard 20-request rejection. |
| 7 | Opt-in read cache with TTL | ✅ | `--cache <duration>` root flag (or `M365_CACHE_TTL`, synced via the same `preAction` hook pattern as `--dry-run`) opt-in caches successful Graph `GET` responses on disk under `~/.config/m365-agent-cli/graph-cache/`. Hooked into the single `callGraphUrlWithRetries` choke point in `graph-client.ts` (only `GET`; mutations always bypass it). Entries are keyed by a SHA-256 hash of bearer token + method + URL, so different identities/tokens never share entries; expired entries are pruned opportunistically on writes. `src/lib/graph-cache.ts`. EWS is out of scope (SOAP/POST, not idempotent REST GETs). |
| 8 | `whoami --capabilities` — decode token scopes → usable command groups | ✅ (investigation reversed the premise) | Traced `verify-token --capabilities`: it already fully implements token-scope → per-area read/write decoding via `graph-capability-matrix.ts`. Duplicating that in `whoami` would fork the logic. Instead `whoami` (both Graph and EWS paths, `--json` and text) now carries a `hint`/`Tip:` cross-reference pointing agents at `verify-token --capabilities`. |

## New workload / API coverage

| # | Feature | Status | Notes |
|---|---------|--------|-------|
| 9 | OneDrive/SharePoint sharing & permissions (create/list/revoke links, manage item permissions) | ✅ (gap-filled, not built from scratch) | Investigation found `files`/`word`/`powerpoint` already had create-link (`share`), invite, list/update/remove permission commands; SharePoint sites already had list/update. Filled the actual gaps: (1) `getDriveItemPermission` + `files`/`word`/`powerpoint` **`permission-get`** (`GET .../permissions/{id}`, single-permission fetch was missing everywhere); (2) `shareFile` gained `--expiration`/`--password`/`--no-retain-inherited-permissions`, matching Graph's `createLink` body fields that had no CLI flags before; (3) `sharepoint-client.ts` gained `getSitePermission`/`createSitePermission`/`deleteSitePermission` + `sharepoint` **`site-permission-get`/`site-permission-create`/`site-permission-delete`** (only list/update existed at the site level). All Graph endpoints verified against Microsoft Learn docs before implementing. |
| 10 | Bulk mutation commands (ID-list / `--filter` driven) for mail/todo/planner | ✅ (todo + planner; mail scoped out) | `src/lib/graph-bulk.ts`: `applyBulkGraphRequests()` builds one `$batch` sub-request per target id and runs them through #6's `graphBatchAll`, normalizing per-id `{ok, status, error}` outcomes; `parseBulkIdListOrExit()`/`printBulkOutcomeSummary()` share the `--ids <csv>` / `--json-file <path>` convention already used by `presence bulk`. Wired into **`todo bulk-complete`** / **`todo bulk-delete`**, and **`planner bulk-complete-task`** (a two-round batch: batched GETs to fetch each task's required `@odata.etag` for `If-Match`, then batched PATCHes). `mail` bulk ops (e.g. bulk mark-read) were scoped out: `mail`'s EWS/Graph dispatch (`mail.ts` + `mail-graph.ts`) is the most complex, heavily-tested command in the CLI, and squeezing bulk flags into its existing flat (non-subcommand) structure carried more regression risk than the other two systems' clean subcommand architecture — left as a follow-up using the same `graph-bulk.ts` helper. |
| 11 | `findtime` multi-attendee scheduling depth (working hours, timezone, free/busy in one call) | ✅ | Found `findMeetingTimes`'s `MeetingTimeSuggestion.attendeeAvailability` was already being fetched from Graph (via `returnSuggestionReasons: true`) but silently discarded by `findtime` (unlike the sibling `suggest` command, which already surfaces it) — now included in both `--json` and text output for both Graph strategies (`findMeetingTimes` per-suggestion; the `getSchedule` fallback gets a new top-level `attendeeAvailability` with each attendee's raw `availabilityView`, not just the merged result). Added `--optional <email>` (repeatable; `AttendeeBase.type` already supported `'optional'`, just unwired), `--min-attendee-percentage <n>` (Graph's `minimumAttendeePercentage`, previously hardcoded to 100), and `--timezone <iana>` (previously hardcoded to UTC — `src/lib/timezone-wallclock.ts` converts the search window to the zone's wall-clock time for the Graph request and switches the `Prefer: outlook.timezone` header and working-hours filtering to match, for both Graph strategies). EWS path ignores the three new flags (Graph-only per Microsoft's schedule APIs). Uncovered and fixed a pre-existing Commander shared-singleton option-state leak between `findtime` CLI-integration test cases along the way (added to `resetSharedCommandOptionLeaks()`). |

| 16 | `excel comments-*` `--beta` flag | ✅ (investigation reversed the premise) | Traced `graph-excel-comments-client.ts`: all 5 `comments-*` functions already call `getGraphBetaUrl()` unconditionally — `workbookComment` has no v1.0 equivalent, so they're *always* beta regardless of any flag. A `--beta` flag would have been a **no-op that misleads users** into thinking it's needed/optional. Fixed the actual bug instead: reworded all 5 command descriptions from the ambiguous "requires beta or GRAPH_BETA_URL" to state plainly that they always use the beta root and only `GRAPH_BETA_URL` customizes *which* beta host. Added regression tests locking in the always-beta behavior and the `GRAPH_BETA_URL` override. |

## Working notes

- Branch: `claude/feature-roadmap-implementation`
- Each item lands as its own commit (or small group of related commits), gated by the full
  local suite (typecheck, biome, knip, tests, coverage, graph inventory, permission matrix)
  before commit.
- Items are implemented roughly in table order, but reordered opportunistically when one
  unblocks another (e.g. #1 before #2).
